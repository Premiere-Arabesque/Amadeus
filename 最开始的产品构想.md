# Persona-Driven Agent Runtime —— 项目架构总结

## 一、项目定位

做一个**通用的人格驱动行为模拟引擎**（Persona-Driven Agent Runtime）。

核心能力：输入人设描述 → 输出符合人设的连续真人行为轨迹。

- 输入女友人设 → AI伴侣（虚构角色，情感陪伴）
- 输入自己的资料 → 数字分身（模拟真人行为）
- 底层是同一个引擎，不同输入对应不同场景

对标关系：OpenClaw是task-driven agent runtime，本项目是character-driven agent runtime。

## 二、学术定位

**Generative Agents论文的产品化落地——从虚拟沙盒走向真实互联网，从多agent社会模拟走向单agent亲密关系。**

填补的gap：
- 2023 Generative Agents：有行动轨迹但人格浅 + 封闭沙盒
- 2024 Generative Agent Simulations of 1,000 People：有深度人格但没有行动轨迹
- **本项目：深度人格驱动 + 真实互联网 + 连续行动轨迹**

核心学术创新点：
1. 概率化行为决策：用logprobs把agent决策从确定性变为随机过程
2. 真实互联网行为：通过MCP从封闭沙盒扩展到开放互联网
3. 情感向量检索：在传统语义检索基础上增加情感维度，基于认知心理学的心境一致性记忆效应
4. 自上而下的细节补全（Detail Elaboration）：现有反思机制是自下而上（观察→反思），本项目增加自上而下方向（计划→细节补全）
5. 多模型分层架构：对话/决策/杂活分离，成本和效果双优化

## 三、整体运行流程

### 3.1 角色初始化

1. 用户提供初始人设文案（粗略或详细均可）
2. AI发起**访谈式深度追问**，通过多轮对话挖掘角色细节（不是填标签，而是讲故事）
3. 访谈产出写入soul.md，作为角色的完整人格描述
4. 对soul.md进行**Expert Reflection**（参考2024论文）：让LLM分别以心理学家、社会学家、行为经济学家等视角生成多角度反思，补充soul.md

### 3.2 日计划生成

每天00:00或角色起床时，触发cron任务：

1. **日计划**：基于soul.md + 核心记忆 + 近期活跃记忆，生成当天要做的事及顺序
2. **小时级计划**：递归分解日计划为每小时的行动安排
3. **分钟级计划**：进一步分解为5-15分钟粒度的具体动作，每个动作带预估时长

（复用Generative Agents论文的三层递归分解机制，但计划本身是概率性的"倾向"而非确定性时间表）

### 3.3 心跳执行循环（Heartbeat Loop）

每**5分钟**一次心跳（参考OpenClaw heartbeat机制，但从task-driven改为persona-driven）：

1. 检查当前分钟级计划是否仍在执行中（惯性机制：每个行为有预估时长，时长内默认继续）
2. 如果当前行为时长到了，用**logprobs采样**决定下一个行为类型
3. 执行行为（见3.4行为执行）
4. 行为完成后更新**情绪状态**
5. 用logprobs判断是否需要**replan**

### 3.4 行为执行（三区模型）

**强真实区（Real）**：通过MCP真实执行
- 刷小红书/B站/微博（MCP拉真实内容）
- 搜索信息、看新闻
- 给用户发消息、分享内容
- 产生真实数据和记忆，追问细节经得住

**弱真实区（Narrative-Deduced）**：从人设推导合理细节
- 通勤、吃饭、日常routine
- soul.md里有"住在五道口，在中关村上班"→ 通勤细节自动生成
- 不需要真实执行但细节自洽

**模糊区（Narrative-Elaborated）**：叙事 + 自我QA补全细节
- 做实验、上课、工作等专业行为
- 行为完成后触发**Detail Elaboration**：agent自问自答补全细节
- 细节写入记忆，跨时间积累形成连贯的经历链条
- 细节深度取决于人设信息量

**MCP Fallback机制**：工具调用失败时，自动退化为叙事行为而不是报错（真人也会"算了不刷了"）

### 3.5 Replan机制

两种触发条件：
1. **每个分钟级计划完成后**：根据执行结果，用logprobs判断是否需要调整后续计划
2. **每次用户消息介入后**：对话结束后，用logprobs判断是否需要调整后续计划

本质：每次与外界交互（执行操作/用户消息）后都考虑replan。

### 3.6 用户消息处理

用户随时可能发消息，处理流程：
1. 中断当前行为（可带延迟，真人也不秒回）
2. 检索活跃记忆，获取相关上下文
3. 结合核心记忆（人设+关系+情绪），生成回复
4. 回复后触发replan判断
5. 恢复或调整行为计划

### 3.7 情绪状态模块

每个分钟级计划完成后更新一次，**双重表示**：
- **文本表示**：喂给对话引擎，让Claude生成符合当前情绪的语气
  - 例："有点低落，因为实验不顺利，但刚刷到一个有趣的视频稍微好了一点"
- **VAD向量表示**：Valence（效价）/ Arousal（唤醒度）/ Dominance（支配感），三个浮点数
  - 直接对接情感向量检索模块

情绪被行为结果、记忆检索、用户互动共同影响，反过来影响决策概率和回复风格。

## 四、记忆系统

### 4.1 四层记忆架构

**原始记忆（Raw Log）**
- 完整未加工数据：对话原文、MCP返回的完整内容、行为执行日志
- 不参与检索，append-only日志文件，按天存储
- 作用：可追溯性、论文评估数据源、记忆重建的原材料

**核心记忆（Core Memory）**
- 常驻prompt，每次LLM调用都带着
- 内容：人设摘要、关系状态、当前情绪（文本+VAD向量）、今日计划摘要
- 不做检索，被其他层的事件驱动更新
- 相当于："我是谁，我现在怎么样"

**活跃记忆（Active Memory）**
- 近期（7-14天）的加工后记忆条目
- 每条带：内容、时间戳、类型（observation/reflection/plan/detail）、importance score、语义embedding、情感embedding
- 三重检索（语义向量 + BM25 + 情感向量）+ reranker精排
- 相当于："最近发生了什么"

**归档记忆（Archive Memory）**
- 活跃记忆过期后压缩的摘要
- 保留关键事件和高importance条目，细节丢弃
- 平时不参与检索，活跃记忆检索不到时回退触发
- 可通过时间戳定位到原始记忆重建完整细节
- 相当于："以前发生过什么"

**数据流向：** 外部输入 → 原始记忆 → 提取加工 → 活跃记忆 → 压缩 → 归档记忆，同时活跃记忆中的关键事件持续更新核心记忆。

### 4.2 三重检索 + 精排

- **语义向量检索**：基于内容语义相似度（标准embedding模型）
- **BM25检索**：基于关键词匹配（精确召回）
- **情感向量检索**：基于情感相似度（emotion embedding模型），实现心境一致性记忆效应
- 三路召回合并去重后，经**Reranker精排**（需考虑情感权重）

### 4.3 记忆衰减与整合

- 每日凌晨日计划生成前，定时任务压缩过期活跃记忆为摘要
- 高importance记忆保留更长时间
- 原始记忆永不删除

### 4.4 可插拔设计

记忆模块通过统一接口（MemoryStore）抽象，底层存储可替换：
- 第一版：SQLite + 本地faiss
- 后续可换为Qdrant、Milvus等向量数据库
- 接口定义先行，具体实现分阶段迭代

## 五、多模型分层

| 模型层 | 用途 | 选型 |
|--------|------|------|
| 对话引擎 | 人格表达、情感回复 | Claude（灵魂） |
| 决策引擎 | logprobs概率化决策、replan判断 | DeepSeek/OpenAI兼容模型（便宜+logprobs） |
| 杂活引擎 | 内容摘要、记忆提取、Detail Elaboration、情绪更新 | Haiku/最便宜模型 |

## 六、系统架构

```
Python Daemon（持续运行）
├── 心跳调度器（每5分钟触发，asyncio）
├── 日计划生成器（每日cron）
├── 决策引擎（DeepSeek logprobs）
├── 对话引擎（Claude API）
├── 行为执行器
│   ├── MCP Client（强真实区：真实执行）
│   ├── 叙事生成器（弱真实区：推导细节）
│   └── Detail Elaborator（模糊区：自我QA补全）
├── 情绪状态模块（文本+VAD双重表示）
├── 记忆模块（可插拔）
│   ├── 原始记忆（append-only日志）
│   ├── 核心记忆（常驻上下文）
│   ├── 活跃记忆（三重检索+精排）
│   └── 归档记忆（压缩摘要）
└── 消息通道（可扩展）
    ├── Web端（第一版：Python后端 + HTML前端）
    └── 后续：Telegram / Discord / 等
```

## 七、部署方案

- 后端打包成exe（PyInstaller），内置cloudflared自动分配临时公网地址
- 用户体验：下载exe → 双击 → 填API key → 扫二维码 → 手机浏览器聊天
- 不需要备案、不需要应用商店、不依赖第三方平台
- 完全本地运行，数据自主

## 八、评估方案

- **行为可信度**：盲测，让用户猜主动消息是AI还是真人发的，正确率越接近50%越好
- **情感连接度**：用户主动聊天频率、收到主动消息后的回复率、长期留存天数
- **对比实验（Ablation）**：
  - logprobs决策 vs LLM硬输出true/false
  - MCP真实内容 vs web search摘要
  - 情感向量检索 vs 纯语义检索
  - Detail Elaboration vs 无细节补全
  - 完整架构 vs 各组件消融
