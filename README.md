# Amadeus

> 一个面向角色扮演场景的行为拟真 Agent Runtime，探索 AI 角色如何从“说话像人设”走向“像人设一样生活”。

Amadeus 是一个仍在开发中的角色扮演 Agent 原型。传统 Roleplay 多关注回复是否符合人设，而这个项目更关注：Roleplay Agent 能否根据角色设定，在连续时间、外部事件、长期记忆、真实工具反馈与模拟场景中行动、经历并改变后续行为。

项目当前围绕一条完整闭环展开：

```text
Persona -> Planning -> Execution -> Memory -> Replan -> Interaction
```

核心想法：

- **从语言拟真到行为拟真**：角色不只需要“说得像”，还需要能生成计划、执行行动、获得场景反馈、沉淀记忆，并让经历影响后续行为。
- **Roleplay Agent + Executor Agent 分层**：Roleplay Agent 全程只用自然语言表达意图；Executor Agent 负责把动作解释为真实工具调用或模拟场景反馈，减少 tool schema、函数调用等内容进入角色上下文造成的出戏感。
- **真实 / 模拟执行平滑路由**：已注册能力自动进入真实工具或 MCP 调用；未注册能力平滑降级为模拟场景反馈，使角色生活流在工具覆盖不完整时仍能保持连续。
- **生活流驱动的主动互动**：主动联系用户不是随机定时唤醒，而是从计划执行、场景反馈、记忆召回和 replan 中自然涌现。

## 项目定位

Amadeus 是一个面向“可持续生活的虚拟人格”的 agent 项目。

它不是单轮聊天机器人，而是一个可以持续运行的人格驱动行为系统：基于用户设定的人设长期存在，按天生成计划，按分钟推进动作，通过消息渠道与外界互动，必要时调用 MCP 做真实操作，并在执行结果或外界介入后持续 replan。

一句话概括：

`输入人设描述 -> 输出符合该人设的连续生活轨迹与互动行为`

这个引擎既可以承载虚构角色，也可以承载真实人的数字分身。底层是同一个 runtime，只是输入的人设和场景不同。

## MVP设计原则/边界

当前已经确认的高层原则：

- 先支持 `1 个角色`
- 先支持 `1 个真实用户`
- 先接入 `1 个消息渠道`
- 先做 `MCP 兼容层`
- 先做一版记忆架构
- 先保证自然 fallback，不追求第一版就覆盖全部复杂能力
- 系统优先保证：`连续`、`自洽`、`可回溯`、`可扩展`

第一阶段先收敛到以下范围：

- `1 个角色`
- `1 个真实用户`
- `1 个消息渠道`
- `1 个最小可用 MCP 能力`
- `1 套完整运行闭环`

这个闭环至少包括：

- 人设初始化
- 分钟级动作执行
- 执行后 replan
- 记忆写入
- 对外消息回复


当前按 README 收敛 MVP 时，可优先保证：

- agent 自己的生活主链路能跑起来
- 用户能在任意时刻介入并影响后续计划



### MVP 运行驱动说明

在当前 README 约定下，MVP 是 `事件驱动` 的，而不是 `heartbeat 驱动` 的。
- MVP 暂不实现“空闲时每隔几分钟自己想点什么做”的 heartbeat 兜底机制
换句话说：
- 有明确分钟级计划时：由计划时间表推进
- 无明确动作时：系统可以保持安静，直到下一个计划节点或外部事件
这个项目的主体就是roleplay agent,他是我们模拟的主体,什么核心记忆啊,检索记忆啊,都是模拟这个roleplay agent人格以及回忆的动作,然后执行计划是模拟这个roleplay agent真正在生活,但是由于防止他出戏,所以我们让executor把他的自然语言翻译成动作去真正执行,不然就得让roleplay agent输出tool call什么的,还有一些明显出戏的内容,比如schema等等,然后没法执行的就采用模拟的方式。最后把结果包装成scene和result返回。
## 规划策略

MVP 明确采用分层规划，但不做全天分钟级一次性铺满。

当前策略：

- `00:00` 或首次启动时生成“今日摘要计划”
- 每次只对最近的一个计划进行小时计划展开(即队列最左边的那个)
- 每次只展开未来 `5-15 分钟` 的动作

执行语义：

- 小时计划用于限定当前时间窗的方向
- 分钟级动作用于真实执行与调度
- 调度器按下一个分钟级动作的计划时间点唤醒
- 不采用固定间隔 heartbeat 轮询来推进整条生命流

这样做的好处(也许?)：

- 减少长期计划漂移
- 降低成本
- 更贴近真实人的行为节奏
- 更容易在用户介入后自然 replan


## 四大层架构

Amadeus 当前最适合按四个大层来理解，而不是按 service 平铺。

### 1. 角色层

负责回答：`这个人是谁`

关注长期稳定的身份一致性，而不是短期行动安排。

包含内容：
- 人设文案
  - 用户在此设定初始人设文案
  - 通过 AI 追问补全人设设定(追问具体实现参考C:\Users\lenovo\Desktop\agent\论文以及相关资料\2411.10109v1.txt)
  - 最终文案需包括:角色稳定特质,长期偏好,等等
- 背景知识(AI根据人设文案依次生成现成答案让用户选,用户也可以根据自己需要自定义)包括但不限于:
  - 长期关系设定
  - 大概作息时间表(比如学生,那么应该有个大概的课程表。如果是上班族,那么应该也有个大概的上班时间表)
  - 大概活动范围

这一层包含了人格相关信息,以及这个人的大概的社会关系,活动范围,作息时间,最终产物为soul.md（agent的人格根基）

### 2. 生活运行层(运行层)

负责回答：`这个人今天怎么活`

这是系统心脏，负责把角色从设定推进为持续运转的生活过程。

包含内容：

- `Planing`
  - 分为小时计划,分钟计划  
  - 每天0:00点会生成今天的小时计划
  - 先生成小时计划
  - 再分解小时计划生成可执行的分钟计划
  - 每次只把未来即将要到达小时计划展开为结构化数据
- `Replan`
  - 在每次执行器执行完毕生成执行结果、外界消息介入后会有一个Replan判定,先进行一次记忆检索,然后将结果和核心记忆以及这次执行结果送给大模型,根据其输出的logprobs的概率值进行判定决定是否replan,以及replan的理由和级别。如果判断为replan,则重新调用一次大模型,让其进行上面的planing操作,但是起始时间点为当前时间。
  
- `Execution`
  - tool包含内部工具(比如假如要发表情包应该调用的是内部的tool),mcp服务,api等等,统称为tool 
  - 执行分钟级动作(只有分钟级动作才会真正被执行)
  - 支持 `Real zone/Weak Real Zone/Ambiguity Zone` 三种执行模式
  - 执行结束后产出标准化outcome
  - 实现上建议做成统一执行入口
  - 执行语义分为三个区域：
  - 真实区(Real Zone)——有对应的tool能真实执行的行为。刷小红书、搜索信息、看新闻、给用户发消息。tool返回真实数据，agent获得真实的体验和记忆，追问任何细节都经得住。

  - 弱真实区(Weak Real Zone)——没有对应tool但可以从人设推导出合理细节的日常行为。通勤、吃饭、洗澡、睡觉。soul.md里写了”住五道口在中关村上班”，那通勤细节可以自动推导出来，不需要真实执行但细节自洽。

  - 模糊区(Ambiguity Zone)——没有对应tool且人设信息不足以直接推导细节的专业行为。做实验、上课、写代码。行为完成后触发Detail Elaboration——由执行层模拟出结果返回给agent,然后agent再根据结果采取下一步行动,直到补全细节。

  - tool失败时的fallback——真实区的行为如果tool调用失败，自动退化成弱真实区或模糊区的模式，不报错。

  - 路由机制
    - 由executor来判断属于哪个区,并返回对应的结果
  
  - `ExecutionResult`
    - 不管是哪个区的执行，上层都只拿统一结果
    - 结果正文保持自由文本，便于自然描述
    - 外层保留最小结构壳，至少包含：
      - `status`：例如 `success / failed / fallback`
      - `source`：例如 `Real zone / Weak Real Zone / Ambiguity Zone`
      - `content`：自由文本，描述具体发生了什么
      - `raw_data`：如果有tool原始返回，则保留在这里并进入 Raw Log
  
  - 这样做有两个直接好处：
   - 上层负责角色扮演的agent完全不用感知真实区 / 弱真实区 / 模糊区的内部路由
   - 同一个动作在后续接入新tool之后，可以自动从模拟升级为真实执行，而不需要改主逻辑
- `Execution loop:`
例子:假如现在有个分钟计划是"刷小红书",然后我们已经有小红书的mcp注册进了tool列表,所以会被路由到强真实区,然后该分钟级计划进入Execution loop开始执行:
1. executor先调MCP拿到真实内容
2. 把内容包装成自然场景描述返回给上层负责角色扮演的agent：
   "你现在躺在床上刷小红书，推荐流里出现了这些内容：
    1. 一篇秋季穿搭指南，博主推荐焦糖色大衣
    2. 学校附近新开了一家日料店
    3. 一个很可爱的猫咪视频
    你会怎么反应？"
3. 负责角色扮演的agent输出自然语言反应：
   "穿搭那个看看...焦糖色大衣挺好看的。
    等等那个日料店，男朋友之前说想吃日料！存一下。"
4. 然后executor再根据负责角色扮演的agent的输出,用tool真实执行相关操作,之后又将返回的结果包装成自然场景描述返回给上层负责角色扮演的agent,以此循环往复,直到终止条件触发。
5. 终止条件:
   1. 负责角色扮演的agent输出中不再产生新的可执行动作(自然终止),比如某一轮只输出了:"猫咪好可爱哈哈",那么executor则返回一个固定字段,比如"loop stop",此时loop 终止(其实原理和tool-calling loop原理一样,模型不输出tool-calling则loop 终止)
   2. 达到最大轮次
   3. 因为每次执行完一次分钟级计划都要进行replan判定，所以在下个分钟级计划开始执行的时间到来之前应该有个预留的buffer，这个参数可以随时调整
   4. 外部中断（交互层有消息到达）
   
总结:executor实际上是一个agent,负责将进行角色扮演的agent输出的自然语言转化为真实操作通过tool执行,将返回的结果包装成自然场景描述返回给上层负责角色扮演的agent,这样负责角色扮演的agent记忆里以及上下文就不会出现任何schema格式的tool calling指令,保持了角色人设避免出戏,并且Execution loop和tool-calling loop的逻辑非常相似。




弱真实区(同理,只是由于此时没有tool可以使用,executor用便宜模型基于人设和推断出的当前场景模拟出结果假如现在有个分钟计划是"去食堂吃饭")：

1. 返回"你去食堂吃饭，今天的菜有番茄鸡蛋、红烧排骨、麻婆豆腐"
2. 负责角色扮演的agent输出自然语言反应:"要了份番茄鸡蛋，食堂今天人好多"
3. 再重新返回模拟后的结果,以此循环往复直到达终止条件

模糊区(同理,只是模拟出的细节更加详细,并且负责角色扮演的agent执行之前需要检索之前相关的记忆来保持连续性——"昨天实验loss是0.8，今天应该在这个基础上继续")




#### 一天总体流程的例子:
1. 当天0:00点会带着昨天结束那一刻的RoleplayAgentContext并进行一次检索操作(检索时过滤掉昨天的记忆因为已经在上下文里,昨天的RoleplayAgentContext仅生成计划时用,并不会代入下一天),然后开始生成今天的小时计划(注意起始时间当然不会是0点,只是0点提前生成了而已,你应该懂我意思)
2. 生成plan之后,然后到点了就根据plan进入execution层执行,每次执行执行完毕之后然后将完整双loop记录(就是RoleplayAgent输出的内容和executor返回的内容)写回RoleplayAgentContext
3. 所有plan执行完毕,一天结束,角色进入睡眠状态

execution loop 每一轮 executor 返回内容后。 query： executor 返回的内容 
和用户对话时。 query：用户发来的消息。
检索到条目之后,以
"你想起了一些事情：

"

- `Planning例子(其实Replan原理也一样,只是起始时间变了):原则：只有程序需要解析的地方才要求结构化，其他全部自由文本。` 
  - 小时计划  → 自由文本（人读的，程序只需要按顺序遍历）

    也可以直接一步生成json,目前采用一步生成json的方案,获得分钟级计划只需要两次调用(第一次调用直接获得展开之后的日计划-也就是小时计划,第二次调用直接获得展开之后的小时计划-即分钟计划),此时只有小时计划和分钟计划

    小时计划展开（自由文本）：

    prompt: 核心记忆+小樱现在到了19:00-20:30的"刷手机放松"时段。请展开这个时段的具体安排。

    output（自由文本）:
    先躺床上刷一会小红书，看看有没有好看的穿搭和美食。
    刷半小时左右换B站看看有没有更新的番剧。如果没什么
    好看的就刷刷微博热搜看看今天发生了什么。中间可能
    去倒杯水或者吃个零食。差不多到9点半就别刷了，准备
    跟男朋友聊天。


    分钟计划展开（结构化JSON，程序要用）：

    prompt: 请将以下安排展开为具体动作序列。
    输出JSON数组，每个动作包含description和duration_minutes。

    output（结构化，程序要解析）:
    [
      {"description": "躺床上打开小红书刷推荐流", "duration_minutes": 15},
      {"description": "看到一篇关于秋季穿搭的帖子，仔细看看", "duration_minutes": 5},
      {"description": "继续刷小红书", "duration_minutes": 10},

    ActionExecutor拿到每个动作后的路由：

    "躺床上打开小红书刷推荐流" → 匹配到小红书MCP → 真实执行，拉回3条推荐内容
    "看到一篇关于秋季穿搭的帖子" → 上一步MCP已返回真实内容，基于真实内容生成反应
    "切到B站看看推荐" → 匹配到B站MCP → 真实执行
    "去倒杯水" → 无MCP匹配 → 叙事生成（弱真实区，直接一句话描述）
    "刷微博热搜" → 匹配到微博MCP → 真实执行

    男朋友之前说想吃日料，存一下回头跟他说。


    总结一下就是：规划层面让LLM自由发挥不约束，执行层面要求结构化让程序能解析，

总结:这一层负责产出“经历”和“行为结果”的原始数据，也就是系统最原始的一手运行数据。


### 补充：状态快照（Snapshot）

仅保存 Raw Log 适合追溯“发生了什么”，但不一定能完整还原“系统当时处于什么状态”。

因此建议同时保存关键状态快照，例如：

- 当前 Core Memory
- 当前计划状态
- 当前推断缓存（如果有）
- 当前运行时队列或任务状态

可以理解为：

- `Raw Log = 事实流`
- `Snapshot = 状态面`

两者结合，才能更完整地回溯系统运行过程。


### 3. 交互层

负责回答：`这个人怎么和外界接触`

重点不是聊天本身，而是把外部世界的介入转成系统内部可处理的运行时事件。

包含内容：
- `Communication_layer`
  - 统一通信层，而不是单纯对话层
  - 负责 `user -> agent`
  - 负责 `agent -> user`
  - 后续支持 `agent -> agent`
- `Channels`
  - 具体渠道适配器，例如 QQ、Telegram、Web 等
- `Interaction`
  - 目前MVP阶段仅支持单用户和agent进行交流(后续会有agent之间互相交流,以及多用户和一个agent交流),并且每次收到用户的信息,agent都会立即停下正在执行的事情,并保存正在执行的事情已经获得的结果,然后带着核心记忆与用户交互。交互完毕后,会保留完整的原始交互数据。
  - 和用户交流的时候,都会先触发记忆检索,再输出答案
  - 交互层产生的记忆都会标记来源(即互动对象的名字),方便后续和同一对象互动时精准检索
  - 主动触达的实现:当roleplay agent的输出中有明显想联系用户或者其他角色(agent to agent先不做)的意图时,executor 会返回一个特殊字段,此时execution loop结束,进入交互状态
  - 需要维护一个角色表,主动触达只会对被注册的角色发起(包括用户和其他agent)
  
每次用户发来消息，触发一次 roleplay agent 调用，输出回复。系统发送回复后，进入"对话冷却期"——比如 3-5 分钟。
冷却期内：

用户继续发消息：直接触发下一次 roleplay agent 调用，对话继续。冷却期重置。
用户没有新消息：系统什么都不做，角色处于"正在等待回复"的状态。

冷却期结束后：

还是没有新消息：系统认为这次对话告一段落(可以理解为execution层的动作执行完毕)，角色进行一次replan,然后继续过自己的日子。

这个机制的几个好处：
没有"明确判断结束"的逻辑负担。 不需要检测 roleplay agent 的输出里有没有"再见"、"晚安"，也不需要用模型去判断对话是否结束。冷却期是一个简单的时间机制，几行代码搞定。
符合真人体验。 真人聊天就是这样——你回复完一条消息后，可能继续聊也可能就此搁置。角色也应该有这种"我说完了，等你"的状态，而不是机械地等到某个信号才"退出对话模式"。
给主动触达留了空间。 冷却期之外的时间，角色回到 execution loop，可能在某个时刻基于当下场景想起用户、主动发消息。这就和主动触达机制无缝衔接了。

几个需要细化的点：
冷却期长度怎么定？ 我建议初版用 3-5 分钟。太短了用户稍微想一下再回复就被判定为"新对话"，体验割裂。太长了角色一直被"占用"，没法回到 execution loop。3-5 分钟是真人聊天里"正常思考回复"的时间窗口。

这一层把用户消息、渠道事件、主动外发消息统一

### 4. 基础设施层

负责回答：`系统怎么稳定落地`

它不定义角色行为，但决定系统是否可运行、可扩展、可回放、可维护。

包含内容：

- `runtime_orchestrator`
  - 系统总调度器
  - 负责事件队列、计划推进、模块编排
- `tool注册表`
  - 负责将内部工具以及mcp,api等统一注册为tool
- `mcp兼容层`
  - 负责将mcp注册为tool、让整个项目不脱离mcp生态
- `storage`
  - 持久化、索引、快照、检索底层
- `logging`
  - 系统日志、审计日志、评估日志
- `models`
  - 模型路由、参数管理、调用封装

这里尤其要强调：`runtime_orchestrator` 虽然归在基础设施层，但它是整个系统的控制中枢。

## 横切子系统：记忆系统

记忆系统不适合被理解成“普通存储层”，它更像一个横切角色层、生活运行层和基础设施层的状态系统。

边界定义如下：

- `生活运行层` 负责产出经历
- `记忆系统` 负责把经历变成可追溯、可检索、可回忆、可压缩的记忆


一句话概括：

`运行层负责发生什么，记忆系统负责怎么记住`

### 四层记忆架构

#### 1. 原始记忆（Raw Log）

- 完整未加工数据
- 内容包括：对话原文、tool返回原文、动作执行日志、replan 判断、关键内部事件
- append-only，按天存储
- 默认不参与检索

作用：

- 追溯系统当时经历了什么
- 作为记忆重建和评估的数据源
- 支持 debug、审计、replay、论文分析

#### 2. 核心记忆（Core Memory）

- 常驻 prompt，每次关键 LLM 调用都会携带,只存“不管当前在做什么，都几乎应该一直带着的东西”
- 不做检索，由事件驱动更新

内容包括：

- 角色层产出的soul.md
- 少量长期稳定事实
例如：住在哪里、在读/在做什么、长期身份设定
少量已经沉淀下来的长期关系结论
不是聊天记录本身
而是像“和谁关系亲近/疏远、长期在意谁”
少量被明确提升为“长期重要”的结论
比如长期目标、长期承诺、重要禁忌



它相当于：

`我是谁，我现在怎么样`

#### 3. 活跃记忆（Active Memory）

- 最近 `7-14 天`(mvp阶段由于暂时不实现归档记忆,故超过14天的也暂时定为活跃记忆) 的加工后记忆条目(加工是指让大模型对记忆加工,MVP阶段先不纠结质量,能用就行)
- 每条记录包含：
  - 内容
  - 时间戳
  - importance score(MVP阶段先不做)
  - 语义 embedding(用阿里云的embedding api实现)
  - 情感 embedding
  - 来源标记(仅标记便于区分,agent不可见)：
    - 真实区
    - 弱真实区
    - 模糊区



它相当于：

`最近发生了什么,用于回忆"昨天及之前"的事`
MVP阶段只对Active Memory记忆进行检索:
一次检索流程为：
- 语义向量检索
- BM25 检索
- 情感向量检索
- 最后把前三者检索到的结果汇总,用reranker 精排(用大模型实现,但是设计为开关形式,可以手动打开/关闭)获得top n相关(n,和相关性阈值两个参数都可以随时调整)
  
#### 4. 归档记忆（Archive Memory）(归档记忆MVP阶段先不做,先把系统跑起来)

- 活跃记忆过期后的压缩摘要
- 保留高情感浓度事件和高 importance 条目(importance具体实现待定)
- 平时不参与常规检索
- 活跃记忆召回不足时作为回退层
- 可通过时间戳回指 Raw Log 重建细节

它相当于：

`以前发生过什么`
#### 检索记忆
- MVP阶段一般一次记忆检索是指,从Active Memory中进行检索,检索方法在Active Memory的介绍中已给出。

### 记忆数据流

建议采用以下流向：

`交互层或运行层产生记录->标记不同来源(比如Replan/planing/Real zone等等)归一化为标准事件Raw Log -> 记忆提取与加工 -> Active Memory -> 定期压缩到 Archive Memory(mvp阶段先不做) -> 关键变化回写 Core Memory(mvp阶段先不做)`





## MCP 兼容层职责

`mcp_compat_layer` 是基础设施层里的关键能力层，不是附属工具集合。

MVP 至少要做到：
  - 当前系统有哪些能力可用
  - 把高层动作映射到具体 MCP 工具
  - 上层发统一动作请求
  - 下层不同 MCP 返回统一结果
  - 错误至少归类为：
  - `success`
  - `failure`
  - `time out`






## MVP 技术栈

当前确认的 MVP 技术栈如下：

- `Python 3.12`
- `uv`
- `asyncio`
- `FastAPI`
- `Pydantic`
- `SQLAlchemy + SQLite`
- `JSONL Raw Log`
- `PydanticAI`
- 官方模型 SDK
- 官方 MCP Python SDK
- `pytest`
- `Ruff`

各组件的职责大致如下：

- `Python 3.12 + asyncio`
  - 作为主运行时
- `FastAPI`
  - 提供最小 HTTP API、Webhook、调试入口和后续 Web 渠道入口
- `Pydantic`
  - 定义 `Event / State / Outcome / MemoryEntry` 等核心 schema
- `SQLAlchemy + SQLite`
  - 存储结构化状态、计划、索引、核心记忆元数据
- `JSONL Raw Log`
  - 按天保存 append-only 原始事件流与执行记录
- `pytest + Ruff`
  - 分别负责测试与代码质量

### PydanticAI 的使用原则

当前决定使用 `PydanticAI`，但只把它作为局部能力层，不作为系统总控框架。

也就是说：

- `runtime_orchestrator` 仍然由项目自己实现
- 系统内部标准类型仍然使用自己的 `core/*` schema
- `PydanticAI` 主要用于：
  - 结构化输出
  - 局部 tool call
  - 计划生成
  - 记忆提炼
  - replan 判断

为了便于未来切换框架，必须做好隔离：

- 业务层不直接依赖 `PydanticAI` 的内部类型
- 框架相关实现尽量收敛在 `infra/model_client.py` 等适配层
- `runtime`、`memory`、`persona` 等模块只依赖项目自己的输入输出接口

原则上，未来如果需要替换为别的 agent 框架或直接手写 SDK，改动应主要集中在适配层，而不是波及整个系统。

### 多模型与多 API 路由

MVP 需要从第一版开始支持按工作负载拆分模型配置，而不是默认所有场景共用同一个 API。

至少区分三类角色：

- `dialogue`
  - 面向用户对话
  - 优先表达质量和人格一致性
- `decision`
  - 用于 replan 判断、执行分支选择、策略决策
  - 可以优先考虑成本
- `memory`
  - 用于记忆提炼、压缩、分类、整理
  - 可以优先考虑成本和吞吐

因此系统配置层必须支持：

- 不同角色使用不同 provider
- 不同角色使用不同 model
- 不同角色使用不同 API key 或 base URL

实现上，这层能力应通过项目自己的路由与适配层暴露，而不是把业务代码直接绑定到某个框架或某一个模型供应商。
